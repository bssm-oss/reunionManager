# Reunion Manager MVP Scope Contract

## Product Goal

Reunion Manager is an Android application that helps a user upload a KakaoTalk conversation export, inspect the conversation locally, and generate a structured reunion plan from that conversation with an optional AI provider.

## Locked MVP Interpretation

This repository is currently greenfield. The safest valid MVP is a single Android app that supports one end-to-end flow:

1. Import a KakaoTalk `.txt` conversation export from local storage.
2. Parse and save the conversation on-device only.
3. Show the imported conversation and key metadata in a simple, trustworthy UI.
4. Generate a structured reunion analysis through an internal analysis interface.
5. Work without login, cloud sync, or server-managed accounts.

## Explicit Product Constraints

### Must Have

- Kotlin-based Android application.
- Coroutine-based asynchronous work.
- Simple, calm, trustworthy UI.
- Local-only persistence for imported conversations, provider settings, and generated analyses.
- No login, logout, or member management.
- A Gemini-compatible AI integration path that can be swapped later.
- Project-grade documentation in `docs/`, plus `README.md` and `AGENTS.md`.
- Quality automation with tests and GitHub Actions CI.

### Must Not Have

- User account features.
- Remote database or cloud sync.
- Hidden background upload behavior.
- Hard-coded API keys or bundled secrets.
- Multiple unrelated MVP flows.
- Overstated certainty in analysis copy.

## Supported Input Format For MVP

The first supported input is a KakaoTalk exported plain-text chat file (`.txt`).

The MVP parser is expected to support:

- chat title/header when present
- sender name
- timestamp
- message text
- multiline message continuation when it can be inferred from the export format

If a file cannot be parsed safely, the app must fail with a clear local error state instead of partially corrupting saved data.

## AI Boundary

The app will define an internal analysis provider contract.

- `FakeAnalysisProvider` will support development and deterministic tests.
- `GeminiAnalysisProvider` will implement the first real provider behind the contract.
- Provider configuration is optional and local-only.
- Missing or invalid provider configuration must not block importing or browsing conversations.

## Storage Boundary

All stored data remains on-device.

Planned persisted data types:

- imported conversation summary
- participants
- parsed messages
- analysis results
- provider settings/configuration metadata

## UX Rules

- The app must explicitly say when data stays local.
- The app must clearly show whether AI is configured.
- Analysis results must be framed as guidance, not certainty.
- Empty, loading, success, and failure states must all be intentional and plain-language.

## Architecture Baseline

The MVP will use a single Android app module with clean internal layers:

- `ui`: Compose screens, state holders, and presentation models
- `domain`: use cases and provider contracts
- `data`: Room entities/DAO/repositories, parser, provider implementations

This keeps the initial repo simple while remaining testable.

## Verification Contract

The implementation is not complete until all of the following are true:

1. `./gradlew testDebugUnitTest` passes.
2. `./gradlew lintDebug` passes.
3. `./gradlew assembleDebug` passes.
4. Parser tests cover valid, malformed, and multiline KakaoTalk transcript cases.
5. Repository/provider tests cover local persistence and missing-config behavior.
6. A manual local flow can be executed for import and analysis generation using fixture data and the fake provider.

## Deferred Until After MVP

- Additional import formats beyond KakaoTalk `.txt`
- OCR or screenshot-based ingestion
- Accounts, cloud backup, sync, or collaboration
- Multiple AI providers beyond the initial Gemini-compatible implementation path
- Advanced analytics, dashboards, or visualizations
- Release automation beyond CI validation

## Requirement Mapping

| User Request | MVP Handling |
| --- | --- |
| Kotlin app | Implemented directly |
| Coroutines | Implemented directly |
| Simple trustworthy UI / UX | Implemented directly |
| No login / no member management | Explicitly excluded from MVP |
| Local storage only | Implemented directly |
| Gemini API, but changeable later | Implemented via provider abstraction |
| Detailed files/comments/docs | Implemented directly |
| `docs/`, `AGENTS.md`, `README.md` | Implemented directly |
| CI, lint, tests, functional tests | Implemented directly within MVP scope and must be delivered in the remote repository workflow |
| Branch/commit/PR/self-merge | Required delivery workflow, not deferred |
